# Claude Code 通用工具使用與官方資源擷取指南

## 📋 總覽

本指南提供在處理各大 AI 平台提示工程文檔時的工具使用策略和官方資源擷取方法。適用於本專案中所有 AI 平台的文檔工作，包括 Claude、OpenAI、Google AI 等。

## 🛠️ 工具使用策略

### 🥇 內建工具優先原則

遵循以下優先順序，確保最佳的資料擷取效果：

#### 1. WebSearch（網路搜尋）
- **用途**：搜尋最新的官方文檔和權威資源
- **優勢**：快速、廣泛覆蓋、支援中文查詢
- **範例查詢**：
  ```
  Anthropic Claude prompt engineering 2025 latest documentation
  OpenAI GPT-4 prompting best practices official guide
  Google AI Gemini prompt engineering techniques
  ```

#### 2. WebFetch（網頁擷取）
- **用途**：直接擷取特定官方頁面內容
- **優勢**：精確抓取、格式化內容、適合深度分析
- **適用情境**：已知確切的官方文檔 URL

#### 3. 其他內建工具
- **Read**：讀取本地參考文檔
- **Grep**：搜尋專案內相關內容
- **LS**：瀏覽專案結構

### 🔄 Playwright 備用方案

當內建工具遇到以下情況時，啟用 Playwright：

#### 觸發條件
- 網站阻擋自動化存取
- 需要 JavaScript 渲染的動態內容
- 需要模擬用戶互動才能存取的內容
- 官方網站有存取限制

#### 使用步驟
1. **啟動瀏覽器**：`mcp__playwright__browser_navigate`
2. **截圖確認**：`mcp__playwright__browser_snapshot`
3. **內容擷取**：使用適當的點擊、滾動等操作
4. **資料提取**：`mcp__playwright__browser_evaluate`

### 🔍 資料驗證流程

確保擷取內容的準確性和時效性：

#### 三層驗證機制
1. **官方文檔確認**：直接從官方來源擷取
2. **新聞資源交叉驗證**：搜尋相關技術新聞
3. **權威部落格補充**：參考知名技術部落格的最新分析

#### 時效性檢查
- 確認文檔發布/更新日期
- 搜尋最新版本資訊
- 比對不同來源的一致性

## 📝 內容擷取與整理原則

### ✅ 允許的內容
- **提示技巧**：官方推薦的提示撰寫方法
- **最佳實踐**：官方建議的使用準則
- **範例模板**：官方提供的對話範例（去除程式碼）

- **程式碼範例**：所有 API 呼叫、程式實作
- **自創提示詞**：非官方來源的提示範例
- **推測性內容**：未經官方確認的功能或特性
- **第三方工具**：非官方開發的輔助工具

### 📋 內容格式要求
- **出處標記**：每個段落都要標註官方來源連結
- **更新時間**：記錄擷取時間和原文檔更新時間
- **版本資訊**：標明適用的模型版本或 API 版本

## 📝 文檔撰寫規範

本章節定義專案中所有文檔的撰寫標準，採用微軟文檔風格並保持乾淨的教學導向。

### 🎯 核心原則

#### 微軟風格基礎
遵循微軟官方文檔指南的核心理念：

- **溫暖友善的語調**：如同與另一個人一對一對話
- **簡潔明瞭**：使用簡短句子，讓讀者快速理解和應用
- **以任務為導向**：明確確定讀者身分和目標任務
- **一致性用詞**：一個概念使用一個詞彙，避免同義詞混用

#### 教學文檔特色
- **乾淨簡潔**：避免花俏裝飾，專注於實用內容
- **漸進式結構**：從基礎概念到進階應用的邏輯順序
- **實例導向**：每個技巧都提供可直接應用的範例
- **無障礙友善**：適合不同技術背景的讀者

### 📋 Markdown 格式規範

#### 標題結構
```markdown
# 文檔標題 (H1 - 與檔案名稱對應)
## 主要章節 (H2)
### 子主題 (H3)
#### 具體細節 (H4 - 最深層級)
```

**標題規則**：
- 使用 sentence case（首字母大寫，其餘小寫）
- 標題簡潔有力，清楚表達章節內容
- 避免超過四個層級的標題結構

#### 文字格式標準

| 格式 | 用途 | 範例 |
|------|------|------|
| **粗體** | 重要概念、關鍵詞 | **提示工程** |
| *斜體* | 強調、引用術語 | *自然語言處理* |
| `代碼` | 技術術語、指令、檔案名 | `WebSearch` |
| > 引用 | 官方說明、重要提醒 | > 來自官方文檔的重要提醒 |

#### 程式碼區塊
```markdown
# 正確格式：使用三個反引號並指定語言
```bash
WebSearch: "Claude prompt engineering 2025"
```

# 錯誤格式：缺少語言標記
```
WebSearch: "Claude prompt engineering 2025"
```
```

#### 清單格式
**無序清單**：
```markdown
- 第一項重點
- 第二項重點
  - 子項目使用兩個空格縮排
  - 保持一致的縮排層級
```

**有序清單**：
```markdown
1. 第一步驟
2. 第二步驟
3. 第三步驟
```

#### 連結和圖片
```markdown
# 連結格式
[連結文字](URL)
[Anthropic 官方文檔](https://docs.anthropic.com)

# 圖片格式（需提供 alt text）
![圖片說明文字](圖片路徑)
```

#### 表格規範
```markdown
| 欄位一 | 欄位二 | 欄位三 |
|--------|--------|--------|
| 內容一 | 內容二 | 內容三 |
| 資料一 | 資料二 | 資料三 |
```

**表格使用原則**：
- 僅在需要快速掃描的表格資料時使用
- 避免在表格儲存格中使用多行內容
- 優先考慮使用清單，因為更易於撰寫和閱讀

### ✍️ 內容撰寫指導

#### 語調和風格
- **對話式語調**：使用「您可以」而非「用戶可以」
- **主動語態**：「使用 WebSearch 搜尋」而非「WebSearch 被用來搜尋」
- **具體明確**：避免模糊詞彙，提供確切的步驟和範例
- **正面表達**：說明「如何做」而非「不要做什麼」

#### 句子結構
- **簡短句子**：平均 15-20 字為佳
- **標準語法**：主詞 + 動詞 + 受詞的基本結構
- **避免複雜子句**：一個句子表達一個概念
- **使用常見詞彙**：避免不必要的專業術語

#### 段落組織
- **一段一主題**：每個段落專注單一概念
- **邏輯順序**：按照讀者的思考流程組織內容
- **過渡自然**：段落間有清楚的邏輯連接
- **適當長度**：避免過長段落，通常 3-5 句為佳
